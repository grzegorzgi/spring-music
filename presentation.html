What makes it hard

Our Strategy — Strangler Fig

Don't rewrite. Don't stop the world. Extract one seam at a time, keep the monolith running, prove each extraction with tests before switching traffic.

Key Decisions

1
Strangler Fig over rewrite — monolith stays in production throughout. Each extraction is validated by the characterization suite before traffic switches. Risk is incremental, not all-at-once. ADR-002
2
TypeScript for the first extracted service — forced type corrections the Java code never caught. releaseYear: string → year: number. The ACL makes the type fix explicit and testable. ADR-003
3
Hook for hard blocks, prompt for soft preferences — the PreToolUse hook deterministically blocks cross-boundary imports. Claude cannot be talked out of it. The CLAUDE.md preference for "add features to services/" is probabilistic guidance. Different guarantees, different tools. ADR-004
4
Characterization tests before everything — no extraction starts until the pin suite is green. When behavior changes unintentionally, the failure names the exact endpoint or field that changed. ADR-001

The Fence — Boundary Enforcement

The problem it solves

Legacy field names (releaseYear, albumId, _class) must never appear in new service API responses. Without enforcement, they leak — silently.


Three enforcement layers

  • Hook — PreToolUse blocks any file write containing a cross-boundary import. Deterministic. Cannot be overridden by prompting.
  • Prompt — CLAUDE.md preference guides where new features go. Probabilistic.
  • Test — fence.test.ts asserts no legacy field name appears in any album-service response shape. Fails with exact field name.
// fence.test.ts
const LEGACY_FIELDS = [
  "releaseYear",
  "albumId",
  "_class"
];

it("no legacy fields in response", () => {
  const response = getAlbum("1");
  LEGACY_FIELDS.forEach(field => {
    expect(
      JSON.stringify(response)
    ).not.toContain(field);
  });
});

If a field slips through, the test names exactly which field. No guessing.

The Scouts — Agentic Seam Scoring

What it does

One Claude API subagent per seam, running in parallel. Each independently scores its candidate on four dimensions and returns structured JSON. A coordinator aggregates into a ranked list.


Scoring dimensions

  • Coupling — how many things depend on this seam
  • Test coverage — how much behavior is pinned
  • Data-model tangle — how messy is the data boundary
  • Business criticality — what breaks if this goes wrong

Result

Scouts agreed with the human architect on 4 of 5 seams. Disagreed on Platform Adapter — scouts rated data-model tangle higher than the human had. Human updated the ADR.

// coordinator.py (simplified)
seams = load_json("seams.json")

with ThreadPoolExecutor() as ex:
    futures = {
        ex.submit(score_seam, s): s
        for s in seams
    }

results = [
    f.result()
    for f in as_completed(futures)
]

ranked = sorted(
    results,
    key=lambda r: r["risk_score"]
)

Context passed explicitly to each subagent — subagents don't inherit coordinator context.

How We Used Claude Code

What worked best

  • Three-level CLAUDE.md — architecture rules taught once, respected every session
  • Characterization tests — Claude read legacy source, generated behavior-pinning tests in one pass including the PUT/POST inversion bug
  • Scaffold generation — full TypeScript service (models, routes, repo, OpenAPI, tests) from Java source in under 30 minutes
  • ADR writing — architect gave seam names, Claude drafted the consequences and tradeoffs

What surprised us

  • The hook made boundary enforcement invisible — Claude simply couldn't write the wrong import, no arguing required
  • Scouts disagreed with human on one seam — and were right
  • CLAUDE.md directory files meant Claude never confused monolith conventions with new-service conventions mid-session

What we'd do differently

  • Set up CLAUDE.md on day one, not after challenges started
  • Write characterization tests before any other code — they became the safety net for everything downstream
